# Ref.BOM_Plastic_Weight

## Назначение

Хранилище ежемесячных снимков веса пластиковых деталей и сборок. Используется для расчёта веса брака по данным карт брака — позволяет взять вес детали **на дату снимка**, а не текущий вес из БОМ. Это исключает ретроактивный пересчёт исторических данных при изменении спецификаций.

---

## Проблема которую решает таблица

Вес пластиковой детали в БОМ может меняться со временем — инженеры корректируют нормы расхода, обновляют спецификации. Если считать вес брака по текущему БОМ, то исторические карты брака (например, за прошлый год) пересчитаются по новым весам, что даст некорректные цифры.

**Решение:** раз в месяц делаем снимок актуальных весов и сохраняем его с датой. При расчёте веса брака джойним по ближайшему снимку, дата которого не позже даты карты брака.

---

| Источник | Что берём |

| `Import_1C.BOM_Wastes` | Нормы отходов (литников) по спецификациям пластиковых деталей |

## Логика расчёта

### Уровень 0 — Первичные пластиковые детали (`#PlasticBase`)

Это самый нижний уровень: детали, которые **непосредственно льются из пластика**.

**Фильтры:**
- Артикул ГП начинается с `8.75.12.*` — это пластиковые детали
- Материал в составе = пластикат (`Nomencl_BigGroupID = 0xB5BD...`)
- Только активные спецификации (`Spec_Status = 1`)

**Расчёт весов:**

| Поле | Формула | Смысл |
|---|---|---|
| `Weight_Total` | `SUM(Porebnost)` | Вес детали + вес литника |
| `Weight_Wastes` | `SUM(QTY из BOM_Wastes)` | Вес литника (отход) |
| `GP_Weight` | `Weight_Total − Weight_Wastes` | Чистый вес пластиковой детали |

> **Важно:** `BOM_Wastes` джойнится предагрегированным (`GROUP BY BOM_ID`) до основного JOIN, чтобы избежать умножения строк когда в БОМ несколько пластикатных компонентов.

---

### Уровни 1–10 — Вышестоящие сборки (`#WorkingTable` + итерации)

Для сборок, в состав которых входят пластиковые детали, `GP_Weight` **прокатывается вверх по дереву БОМ** через итеративный алгоритм (до 10 итераций).

**Инициализация `#WorkingTable`:**
- Для компонентов из `#PlasticBase`: `GP_Weight = plastic.GP_Weight × Porebnost`
- Для всех остальных компонентов: `GP_Weight = 0`
- Только активные спецификации (`Spec_Status = 1`)
- Покупные изделия (артикул `9.*`) исключаются

**Итерация (Шаг 2):**

На каждой итерации:
1. Агрегируем `GP_Weight` по каждой спецификации → `#FilteredSpecs` с `Total_GP_Weight`
2. Находим строки в `#WorkingTable` где `GP_Weight = 0` и у компонента (`Nomencl_ID`) есть спецификация в `#FilteredSpecs`
3. Обновляем: `GP_Weight = Total_GP_Weight × Porebnost`

**Приоритет выбора спецификации** (при нескольких спецификациях на один `GPNomencl_ID`):
1. Pass 1: сначала `Spec_Status = 1`, затем `Start_Day DESC`
2. Pass 2 (резервный): только `Start_Day DESC`

Цикл повторяется пока есть обновлённые строки или не исчерпаны 10 итераций.

---

### Финальный результат (UNION)

| Часть | Откуда | Weight_Total | Weight_Wastes | GP_Weight |
|---|---|---|---|---|
| Первичные детали | `#PlasticBase` | Заполнено | Заполнено | Заполнено |
| Вышестоящие сборки | `#WorkingTable` | `NULL` | `NULL` | Заполнено (сумма пластика в составе) |

> `Weight_Total` и `Weight_Wastes` есть **только у первичных пластиковых деталей**. Для сборок это поля `NULL` — у них нет собственного литника, они просто «содержат» пластик от вложенных деталей.

## Структура таблицы

| Колонка | Тип | Описание |
|---|---|---|
| `ID` | INT IDENTITY | Технический ключ строки |
| `Snapshot_Date` | DATE | Дата снимка (дата запуска процедуры) |
| `Spec_ID` | VARBINARY(16) | ID спецификации из 1С |
| `GPNomencl_ID` | VARBINARY(16) | ID номенклатуры готовой продукции |
| `GPNomencl_No` | NVARCHAR(255) | Артикул готовой продукции |
| `Spec_Status` | INT | Статус спецификации (1 = активная) |
| `Delete_Mark` | VARBINARY(1) | Пометка на удаление из 1С |
| `Start_Day` | DATE | Дата начала действия спецификации |
| `Finish_Day` | DATE | Дата окончания действия спецификации |
| `Weight_Total` | DECIMAL(18,6) | Вес детали + литник (только первичные) |
| `Weight_Wastes` | DECIMAL(18,6) | Вес литника (только первичные) |
| `GP_Weight` | DECIMAL(18,6) | Суммарный вес пластика в изделии |

## Процедура обновления

### `Ref.usp_Refresh_BOM_Plastic_Weight`

Запускается **вручную раз в месяц**. Выполняет полный расчёт актуальных весов и **добавляет** строки в таблицу с текущей датой. Исторические снимки не удаляются.

```sql
EXEC Ref.usp_Refresh_BOM_Plastic_Weight;
```

**Алгоритм процедуры:**
1. Создаёт временные таблицы `#PlasticBase`, `#WorkingTable`
2. Запускает итеративный цикл (до 10 итераций)
3. Делает `INSERT INTO Ref.BOM_Plastic_Weight` из финального `UNION`
4. Проставляет `Snapshot_Date = CAST(GETDATE() AS DATE)` по всем вставленным строкам

### Просмотр доступных снимков

```sql
SELECT Snapshot_Date, COUNT(*) AS Rows
FROM Ref.BOM_Plastic_Weight
GROUP BY Snapshot_Date
ORDER BY Snapshot_Date DESC;
```

## Ручная правка данных

Поскольку расчёт основан на данных 1С, возможны ситуации когда сотрудник указал некорректные данные в справочниках. В этом случае можно вручную исправить или добавить строку в нужный снимок.

**Добавить строку:**
```sql
INSERT INTO Ref.BOM_Plastic_Weight
    (Snapshot_Date, Spec_ID, GPNomencl_ID, GPNomencl_No, Spec_Status, Delete_Mark,
     Start_Day, Finish_Day, Weight_Total, Weight_Wastes, GP_Weight)
VALUES ('2026-03-26', 0x..., 0x..., '8.75.12.XXX', 1, 0x00,
        '2026-01-01', NULL, 15.500, 3.000, 12.500);
```

**Исправить вес:**
```sql
UPDATE Ref.BOM_Plastic_Weight
SET GP_Weight = 12.500
WHERE Snapshot_Date = '2026-03-26'
  AND Spec_ID = 0x...;
```

**Удалить строку:**
```sql
DELETE FROM Ref.BOM_Plastic_Weight WHERE ID = 1126;
```
