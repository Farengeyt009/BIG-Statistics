# Ref.BOM_Stamping_Weight

## Назначение

Хранилище ежемесячных снимков веса металла в штампованных деталях и сборках. Используется для расчёта веса брака по данным карт брака — позволяет взять вес детали **на дату снимка**, а не текущий вес из БОМ. Это исключает ретроактивный пересчёт исторических данных при изменении спецификаций.

---

## Проблема которую решает таблица

Вес штампованной детали в БОМ может меняться со временем — инженеры корректируют нормы расхода металла, обновляют спецификации. Если считать вес брака по текущему БОМ, то исторические карты брака пересчитаются по новым весам, что даст некорректные цифры.

**Решение:** раз в месяц делаем снимок актуальных весов и сохраняем его с датой. При расчёте веса брака джойним по ближайшему снимку, дата которого не позже даты карты брака.

---

| `Import_1C.vw_Nomenclature_Reference_Current` | Справочник номенклатуры — вес единицы материала (`Numerator_Weight`) |

## Логика расчёта

### Уровень 0 — Первичные штампованные детали (`#StampingBase`)

Это самый нижний уровень: детали, которые **непосредственно штампуются из металла**.

**Фильтры:**
- Тип трудозатрат (`Labor_TypeGroupeID`) = один из двух ID групп штамповки
- Только активные спецификации (`Spec_Status = 1`)
- Только компоненты с ненулевым весом (`Numerator_Weight > 0`)

**Расчёт веса:**

| Поле | Формула | Смысл |
|---|---|---|
| `GP_Weight` | `SUM(Numerator_Weight × Porebnost)` | Суммарный вес металла на одну единицу готовой детали |

> В отличие от пластикового расчёта, здесь нет `Weight_Total` и `Weight_Wastes` — только итоговый вес металла. Отходы штамповки (облой и т.д.) в данном расчёте не учитываются.

---

### Уровни 1–10 — Вышестоящие сборки (`#WorkingTable` + итерации)

Для сборок, в состав которых входят штампованные детали, `GP_Weight` **прокатывается вверх по дереву БОМ** через итеративный алгоритм (до 10 итераций).

**Инициализация `#WorkingTable`:**
- Загружаются **все строки БОМ** (не покупные `NOT LIKE '9.%'`, только активные `Spec_Status = 1`)
- Фильтр по `Labor_TypeGroupeID` здесь **не применяется** — он нужен только для первичных деталей
- Для компонентов из `#StampingBase`: `GP_Weight = stamping.GP_Weight × Porebnost`
- Для всех остальных компонентов: `GP_Weight = 0`

**Итерация (Шаг 2):**

На каждой итерации:
1. Агрегируем `GP_Weight` по каждой спецификации → `#FilteredSpecs` с `Total_GP_Weight`
2. Находим строки в `#WorkingTable` где `GP_Weight = 0` и компонент (`Nomencl_ID`) есть в `#FilteredSpecs` как `GPNomencl_ID`
3. Обновляем: `GP_Weight = Total_GP_Weight × Porebnost`

**Пример прокатки через уровни:**
```
Уровень 0 (#StampingBase):  AAA1 → GP_Weight = 2,  AAA2 → GP_Weight = 3
Итерация 1:  BBB1 содержит AAA1(2) + AAA2(3)  →  BBB1.Total_GP_Weight = 5
Итерация 2:  CCC1 содержит AAA1(2) + BBB1(5)  →  CCC1.Total_GP_Weight = 7  ✓
```

**Приоритет выбора спецификации** (при нескольких спецификациях на один `GPNomencl_ID`):
1. Pass 1: сначала `Spec_Status = 1`, затем `Start_Day DESC`
2. Pass 2 (резервный): только `Start_Day DESC`

Цикл повторяется пока есть обновлённые строки или не исчерпаны 10 итераций.

---

### Финальный результат (UNION)

| Часть | Откуда | GP_Weight |
|---|---|---|
| Первичные штампованные детали | `#StampingBase` | Вес металла на единицу детали |
| Вышестоящие сборки | `#WorkingTable` | Суммарный вес металла всех вложенных штамповок |

## Структура таблицы

| Колонка | Тип | Описание |
|---|---|---|
| `ID` | INT IDENTITY | Технический ключ строки |
| `Snapshot_Date` | DATE | Дата снимка (дата запуска процедуры) |
| `Spec_ID` | VARBINARY(16) | ID спецификации из 1С |
| `GPNomencl_ID` | VARBINARY(16) | ID номенклатуры готовой продукции |
| `GPNomencl_No` | NVARCHAR(255) | Артикул готовой продукции |
| `Spec_Status` | INT | Статус спецификации (1 = активная) |
| `Delete_Mark` | VARBINARY(1) | Пометка на удаление из 1С |
| `Start_Day` | DATE | Дата начала действия спецификации |
| `Finish_Day` | DATE | Дата окончания действия спецификации |
| `GP_Weight` | DECIMAL(18,6) | Суммарный вес металла в изделии |

## Процедура обновления

### `Ref.usp_Refresh_BOM_Stamping_Weight`

Запускается **вручную раз в месяц**. Выполняет полный расчёт актуальных весов и **добавляет** строки в таблицу с текущей датой. Исторические снимки не удаляются.

```sql
EXEC Ref.usp_Refresh_BOM_Stamping_Weight;
```

**Алгоритм процедуры:**
1. Создаёт `#StampingBase` — первичные штампованные детали с весами
2. Создаёт `#WorkingTable` — все строки БОМ с инициализированными весами
3. Запускает итеративный цикл (до 10 итераций) — прокатывает вес вверх по дереву
4. Делает `INSERT INTO Ref.BOM_Stamping_Weight` из финального `UNION`
5. Проставляет `Snapshot_Date = CAST(GETDATE() AS DATE)` по всем вставленным строкам

### Просмотр доступных снимков

```sql
SELECT Snapshot_Date, COUNT(*) AS Rows
FROM Ref.BOM_Stamping_Weight
GROUP BY Snapshot_Date
ORDER BY Snapshot_Date DESC;
```

## Ручная правка данных

**Добавить строку:**
```sql
INSERT INTO Ref.BOM_Stamping_Weight
    (Snapshot_Date, Spec_ID, GPNomencl_ID, GPNomencl_No, Spec_Status, Delete_Mark,
     Start_Day, Finish_Day, GP_Weight)
VALUES ('2026-03-26', 0x..., 0x..., '8.XX.XX.XXX', 1, 0x00,
        '2026-01-01', NULL, 5.250);
```

**Исправить вес:**
```sql
UPDATE Ref.BOM_Stamping_Weight
SET GP_Weight = 5.250
WHERE Snapshot_Date = '2026-03-26'
  AND Spec_ID = 0x...;
```

**Удалить строку:**
```sql
DELETE FROM Ref.BOM_Stamping_Weight WHERE ID = ...;
```

## Отличия от Ref.BOM_Plastic_Weight

| | Пластик | Штамповка |
|---|---|---|
| Фильтр первичных деталей | `GPNomencl_No LIKE '8.75.12%'` + `Nomencl_BigGroupID` | `Labor_TypeGroupeID` (два ID) |
| Источник веса | `Porebnost` − `BOM_Wastes` (литник) | `Numerator_Weight × Porebnost` |
| Поля веса | `Weight_Total`, `Weight_Wastes`, `GP_Weight` | только `GP_Weight` |
| Отходы | Литник учитывается (`Weight_Wastes`) | Не учитываются |
